# CoreWar Roadmap & Mentorat Plan

> **Pour Hermes:** accompagner l'utilisateur comme professeur/mentor. Ne pas écrire le projet à sa place sauf micro-exemples pédagogiques. Faire des revues de code à la fin de chaque chapitre avant de passer au suivant.

**Goal:** Construire progressivement un CoreWar propre en C, robuste, testé, extensible, puis ajouter une interface graphique/visualisation en bonus.

**Architecture:** Séparer clairement l'assembleur, le désassembleur/debug tools, la VM, les champions, les tests et plus tard le rendu graphique. Chaque module doit avoir une responsabilité simple, une API C claire, des tests unitaires ciblés et des tests d'intégration avec de petits champions fixtures.

**Tech Stack:** C moderne mais portable (C11 conseillé), Makefile ou CMake, tests unitaires C, Valgrind/ASan/UBSan, clang-format/clang-tidy ou équivalent, puis SDL2/raylib/terminal UI pour la visualisation selon choix futur.

## 1. Principes de travail

### Rôle du mentor

- Je donne la route, les concepts, les pièges, les critères de qualité.
- Tu écris le code principal toi-même.
- Je peux écrire de très petits exemples pour clarifier une idée, mais pas prendre le clavier à ta place.
- À la fin de chaque chapitre, tu m'appelles pour une review : architecture, lisibilité, bugs potentiels, performance, mémoire, tests.

### Rythme recommandé

1. Lire le chapitre.
2. Écrire une petite spécification ou checklist.
3. Implémenter une tranche minimale.
4. Ajouter des tests.
5. Lancer les outils de vérification.
6. Faire une review avec moi.
7. Refactoriser avant de passer au chapitre suivant.

### Définition de “propre” pour ce projet

- Pas de fonctions géantes.
- Pas de variables globales inutiles.
- Pas de parsing mélangé avec la VM.
- Pas de logique métier dans l'affichage.
- APIs explicites entre modules.
- Gestion mémoire documentée : qui alloue, qui libère.
- Tests pour les cas normaux et les erreurs.
- Aucune fuite mémoire connue.
- Messages d'erreur compréhensibles.

## Phase 2 — Comprendre et figer la spec CoreWar

**Objectif:** Écrire ta propre spec avant d'implémenter.

### Documents à produire

- `docs/corewar-spec.md` : règles générales.
- `docs/redcode.md` : syntaxe assembleur que tu supportes.
- `docs/opcodes.md` : table des instructions.
- `docs/architecture.md` : modules et responsabilités.

### Points à clarifier

- Taille de l'arène : souvent `MEM_SIZE = 4096` dans le Corewar Epitech.
- Nombre max de champions.
- Format `.s` source.
- Format `.cor` binaire.
- Endianness utilisée dans les fichiers `.cor`.
- Header champion : magic, name, size, comment.
- Encodage des arguments : registre/direct/indirect.
- Cycles : `CYCLE_TO_DIE`, `CYCLE_DELTA`, `NBR_LIVE`, `MAX_CHECKS`.
- Liste exacte des instructions.

### Instructions classiques Epitech

À documenter :

- `live`
- `ld`
- `st`
- `add`
- `sub`
- `and`
- `or`
- `xor`
- `zjmp`
- `ldi`
- `sti`
- `fork`
- `lld`
- `lldi`
- `lfork`
- `aff`

### Décision importante

Choisir si tu veux reproduire strictement le Corewar Epitech ou faire une version “CoreWar personnelle” compatible mais améliorée. Recommandation :

1. D'abord une version compatible Epitech/Redcode simple.
2. Ensuite seulement, extensions personnelles.

### Critères de fin de phase

- La spec est écrite.
- Les constantes sont listées.
- Les formats binaires sont définis.
- Tu peux expliquer comment une instruction passe de `.s` à mémoire VM.

## Phase 3 — Types fondamentaux et modèle mémoire

**Objectif:** Poser les briques C stables : arena, instruction, champion, process.

### Concepts à maîtriser

- Arène circulaire : toutes les adresses wrap autour de `MEM_SIZE`.
- Lecture/écriture big-endian pour les valeurs multi-octets.
- Différence entre adresse absolue et offset relatif.
- `IDX_MOD` pour certaines instructions.
- Processus VM : PC, registres, carry, wait cycles, alive.

### Modules à créer

- `include/corewar/types.h`
- `include/corewar/arena.h`
- `src/vm/arena.c`
- `include/corewar/instruction.h`
- `include/corewar/process.h`
- `include/corewar/champion.h`

### Fonctions importantes

Arena :

- `arena_init`
- `arena_read_u8`
- `arena_read_i16`
- `arena_read_i32`
- `arena_write_u8`
- `arena_write_i16`
- `arena_write_i32`
- `arena_wrap_addr`
- `arena_load_champion`

Process :

- initialisation PC/registers
- duplication pour `fork`/`lfork`
- avance du PC selon taille instruction

### Tests essentiels

- Adresse négative wrap correctement.
- Adresse > `MEM_SIZE` wrap correctement.
- Lire/écrire 2 et 4 octets en bordure de mémoire.
- Endianness correcte.

### Critères de fin

- Le modèle mémoire est fiable et testé.
- Aucune VM complète encore nécessaire.

## Phase 4 — Assembleur : lexer/parser

**Objectif:** Lire un fichier `.s` et produire une représentation intermédiaire propre, sans encore générer du binaire complet.

### Séparation conseillée

1. Lexer/tokenizer : découpe le texte en tokens.
2. Parser : transforme tokens en AST / liste d'instructions.
3. Résolution labels : calcule les offsets.
4. Encoder : écrit le `.cor`.

### Syntaxe à supporter au début

- `.name "champion name"`
- `.comment "comment"`
- Labels : `start:`
- Instructions : `op arg1, arg2, arg3`
- Commentaires : selon spec choisie (`#`, éventuellement `;`)
- Registres : `r1` à `r16`
- Direct : `%42`, `%:label`
- Indirect : `42`, `:label`

### Structures utiles

- `token_t`
- `parser_t`
- `asm_program_t`
- `asm_instruction_t`
- `asm_arg_t`
- `label_ref_t`

### Erreurs pédagogiques à éviter

- Faire l'encodage pendant le parsing.
- Stocker des pointeurs vers des buffers temporaires.
- Ne pas avoir de position ligne/colonne dans les erreurs.
- Accepter silencieusement une syntaxe invalide.

### Critères de fin

- Parser un champion simple en mémoire.
- Afficher/debugger l'AST.
- Erreurs propres avec ligne/colonne.

---

## Phase 5 — Assembleur : encodage `.cor`

**Objectif:** Transformer l'AST en binaire `.cor` valide.

### Sous-parties

1. Table des opcodes.
2. Calcul de la taille de chaque instruction.
3. Calcul des positions labels.
4. Résolution des références labels.
5. Écriture du header.
6. Écriture des instructions.

### Fonctions importantes

- `op_get_by_name`
- `instruction_encoded_size`
- `arg_encoded_size`
- `program_compute_offsets`
- `program_resolve_labels`
- `cor_write_header`
- `cor_write_instruction`
- `assembler_compile_file`

### Tests essentiels

- `.name` trop long => erreur.
- `.comment` trop long => erreur.
- Label inconnu => erreur.
- Registre invalide => erreur.
- Encodage exact d'une instruction simple.
- Champion fixture compile vers bytes attendus.

### Critères de fin

- Tu peux compiler 3-5 champions simples.
- Le binaire produit est déterministe.
- Tu as des fixtures attendues en tests.

## Phase 6 — Loader `.cor`

**Objectif:** Charger des champions binaires dans la VM.

### Responsabilités

- Lire header.
- Vérifier magic.
- Vérifier taille annoncée.
- Charger le code champion.
- Attribuer un ID champion.
- Placer les champions dans l'arène.

### Fonctions importantes

- `champion_load_cor`
- `champion_validate_header`
- `vm_load_champions`
- `vm_place_champions_evenly`

### Tests essentiels

- Magic invalide.
- Taille incohérente.
- Fichier tronqué.
- Placement de 2, 3, 4 champions.

### Critères de fin

- Une VM peut charger des `.cor` sans exécuter.
- Dump mémoire initial possible.

## Phase 7 — VM : cycle engine minimal

**Objectif:** Faire tourner les processus cycle par cycle, sans toutes les instructions au début.

### Modèle d'exécution

Chaque process :

- lit opcode à son PC,
- attend le nombre de cycles de l'instruction,
- décode les arguments,
- exécute,
- avance ou modifie le PC.

### Fonctions importantes

- `vm_init`
- `vm_step_cycle`
- `vm_fetch_opcode`
- `vm_decode_instruction`
- `vm_execute_instruction`
- `process_schedule_instruction`
- `process_advance_pc`

### Première instruction à implémenter

- `live` uniquement.

### Critères de fin

- La VM charge un champion.
- Le process exécute `live`.
- La VM compte les cycles.
- Un dump/debug montre PC et instruction.

## Phase 9 — VM : règles de vie/mort et victoire

**Objectif:** Implémenter les règles complètes de fin de partie.

### Concepts

- `live` marque un champion/process comme vivant.
- `cycle_to_die` diminue selon `NBR_LIVE`/`MAX_CHECKS`.
- Processus non vivants sont supprimés.
- Le dernier champion ayant dit `live` gagne.

### Fonctions importantes

- `vm_check_lives`
- `vm_kill_dead_processes`
- `vm_update_cycle_to_die`
- `vm_find_winner`
- `vm_run_until_end`

### Tests essentiels

- Champion qui ne live jamais meurt.
- Plusieurs `live` déclenchent réduction de `cycle_to_die`.
- Forked process correctement tué ou conservé.
- Winner stable et déterministe.

## Phase 10 — CLI utilisateur

**Objectif:** Avoir des binaires utilisables.

### Binaires possibles

- `asm` : compile `.s` en `.cor`.
- `corewar` : exécute des `.cor`.
- `disasm` ou option `--disasm` : inspecte un `.cor`.

### Options utiles

Pour `corewar` :

- `--dump N` : dump mémoire au cycle N.
- `--max-cycles N` : sécurité.
- `--seed N` si du hasard est introduit plus tard.
- `--debug` : logs lisibles.
- `-n ID champion.cor` : assigner un numéro.

Pour `asm` :

- `-o output.cor`
- `--debug-ast`
- `--strict`

### Critères de fin

- Utilisation documentée dans `README.md`.
- Messages d'erreur propres.
- Codes de retour cohérents.

## Phase 12 — Visualisation / GUI bonus

**Objectif:** Ajouter une interface visuelle sans polluer la VM.

### Principe clé

La VM ne doit pas dépendre de SDL/raylib/ncurses. Elle expose un état observable, et l'UI le lit.

### Options possibles

1. Terminal UI avec ncurses : rapide, rétro, très CoreWar.
2. SDL2 : plus bas niveau, très contrôlable.
3. raylib : simple et agréable pour affichage 2D.
4. Web UI plus tard : VM exporte snapshots JSON, frontend affiche.

### Interface VM/UI proposée

- `vm_snapshot_t`
- `vm_get_snapshot`
- `vm_step_cycle`
- `vm_run_cycles(vm, n)`
- callback optionnel `on_memory_write`

### Affichage utile

- Arène colorée par champion.
- PC/processus actifs.
- Écritures mémoire récentes.
- Registres du process sélectionné.
- Cycles, cycle_to_die, lives.
- Pause/play/step.
- Vitesse réglable.

### Critères de fin

- L'UI peut stepper la VM.
- La VM fonctionne toujours sans UI.
- Pas de logique CoreWar dans le rendu.

# Méthode de review par chapitre

Quand tu finis une phase, envoie-moi :

1. Le chapitre terminé.
2. Ce que tu penses être propre.
3. Ce dont tu doutes.
4. Les commandes que tu as lancées.
5. Les erreurs ou warnings restants.

Je ferai une review en 4 niveaux :

- **Correctness:** est-ce que ça respecte la spec ?
- **C design:** séparation, headers, ownership, erreurs.
- **Robustesse:** mémoire, edge cases, erreurs utilisateur.
- **Performance:** seulement quand le code est correct.

# Décisions prises au démarrage

1. **Compatibilité / difficulté :** version compatible Epitech en priorité. Les améliorations personnelles viendront seulement après une base fonctionnelle.
2. **Build system :** l'utilisateur est ouvert au meilleur choix. Recommandation mentor actuelle : `Makefile` pour démarrer simplement en C, avec migration possible vers `CMake` si raylib/GUI rend cela utile.
3. **Tests :** utiliser `Criterion`.
4. **UI bonus :** utiliser `raylib` plus tard pour afficher une fenêtre avec des cellules/carrés colorés représentant l'arène.
5. **Binaires :** garder les noms classiques `asm` et `corewar`.
